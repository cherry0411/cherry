## Cherry 项目现有逻辑梳理

### 整体定位
DHT 磁力链接搜索引擎：实时抓取 BitTorrent 网络的 torrent 元数据，提供全文搜索和浏览。

### 数据流（端到端）
```
DHT 网络 (BEP-5/BEP-9)
  → Go 爬虫（6个实例，每个512并发）
  → POST /api/v1/torrents/batch（批量128条）
  → .NET IngestService（Channel 10万容量，批次5000）
  → CuckooFilter 去重（1亿容量，内存+持久化）
  → PostgreSQL BINARY COPY（高速批量写入）
  → 手动脚本全量同步 → Meilisearch 索引
  → Vue SPA 搜索展示
```

### 三个核心亮点
1. **高性能入库**：Npgsql BINARY COPY + 临时表 + ON CONFLICT DO NOTHING，避免行级 INSERT 瓶颈
2. **双引擎搜索**：Meilisearch 优先（支持相关性排序），PG trigram 降级 fallback，CJK 自动切换 `all` 策略
3. **三层去重**：爬虫本地 TTL seenSet → 远程 `/check` API 批量校验 → CuckooFilter 内存过滤

### 已知的潜在优化点
| 问题 | 位置 |
|------|------|
| **Meilisearch 同步非实时**，靠手动脚本全量导入 | `scripts/sync-meilisearch.js` |
| **前端完全无构建工具**，CDN 依赖，无 TypeScript 类型保护 | `frontend/app.js` |
| **`app.js` 仅153行**，四个页面组件都在一个文件里，维护性差 | `frontend/app.js` |
| **统计接口缓存只有10秒**，每次都查 COUNT(*) | `StatsEndpoints.cs` |
| **OutputCache 缓存策略较保守**（搜索15s，详情60s） | `Program.cs` |
| **爬虫到API无认证**，batch 接口完全开放 | `TorrentEndpoints.cs` |

# Cherry DHT 搜索引擎 — 详细设计

## 第一章：整体架构决策

### 技术栈保留

**全部保留，无需替换任何组件：**

| 组件 | 保留理由 |
|------|---------|
| Go 爬虫 | DHT/BEP-5/BEP-9 完整，编码修复已做，无需改动 |
| .NET 10 Minimal API | Channel + IHostedService 完美匹配流式消费场景 |
| PostgreSQL 17 | BINARY COPY 吞吐高，pg_trgm fallback 完善 |
| Meilisearch | 百万文档仅需 200-400MB，中文 unicode 分词原生支持 |
| CuckooFilter | 1亿容量约 200MB，lock-free CAS，远比 Redis 便宜 |
| Vue 3 CDN | 零运行时开销 |

**明确不引入：**
- Kafka / RabbitMQ：2C4G 上额外占 512MB+，现有 Channel(10万) 已满足
- Redis：CuckooFilter 已在内存，统计用原子计数器
- Elasticsearch：内存需求 1GB+，超出预算

### 改进后完整数据流

```
DHT网络
  → Go爬虫（BEP-5/BEP-9）
      ↓ 失败写本地 WAL 文件（新增）
  → POST /api/v1/torrents/batch（X-API-Key 认证，新增）
  → IngestService Channel（超 80% 返回 429 背压，新增）
      → CuckooFilter 去重
      → 批次5000，PostgreSQL BINARY COPY
      → [新增] 触发 MeiliSyncService 异步推送（2s 内可搜索）
      → [新增] 原子计数器 +N
  → Vue SPA 搜索展示

统计路径（改造后）：
  内存原子计数器（每次写入时 Interlocked.Add）
  + 定时每小时 PG reltuples 精确刷新
  → GET /api/v1/stats（O(1)，无 DB 查询）
```

## 第二章：爬虫→API 可靠传输

### 2.1 认证方案（零成本）

静态 API Key via HTTP Header，只保护写入接口：

```csharp
// Program.cs
var apiKey = builder.Configuration["ApiKey"];
if (!string.IsNullOrEmpty(apiKey))
{
    app.Use(async (ctx, next) =>
    {
        static bool IsWritePath(string path) =>
            path.StartsWith("/api/v1/torrents/batch", StringComparison.OrdinalIgnoreCase) ||
            path.StartsWith("/api/v1/torrents/peers",  StringComparison.OrdinalIgnoreCase);

        if (IsWritePath(ctx.Request.Path))
        {
            if (!ctx.Request.Headers.TryGetValue("X-API-Key", out var key)
                || key != apiKey)
            {
                ctx.Response.StatusCode = 401;
                await ctx.Response.WriteAsJsonAsync(new { error = "Unauthorized" });
                return;
            }
        }
        await next(ctx);
    });
}
```

### 2.2 爬虫侧：WAL 本地缓冲（应对网络不稳定）

当前失败直接丢弃。新增基于文件的 pending queue：

```
go/cherry-picker/internal/export/
  ├── wal_sink.go     // 包装 httpSink，失败时写 WAL
  └── wal_replay.go   // 后台 goroutine，每30s 重放
```

```go
// WAL 文件格式：每行一个 JSON batch（按小时轮转）
// wal_2026010215.jsonl → 重放成功后删除

type walSink struct {
    inner  Sink
    walDir string   // /data/wal/
    mu     sync.Mutex
}

func (w *walSink) WriteBatch(ctx context.Context, batch []pipeline.Event) error {
    if err := w.inner.WriteBatch(ctx, batch); err != nil {
        w.appendToWAL(batch)  // 失败写 WAL，不返回 error
    }
    return nil
}

// 后台每30s 扫描 WAL 目录，逐文件逐行重放，成功后删除
// 超过24小时的 WAL 文件自动清除（认为数据已过期）
func (w *walSink) replayLoop(ctx context.Context) { ... }
```

**容量估算**：最坏情况 100% 失败，1小时 ≈ 20MB WAL 文件，完全可控。

| 层次 | 机制 | 说明 |
|------|------|------|

## 第三章：数据入库管道改造

### 3.1 Meilisearch 实时增量推送（核心改进）

新增 `MeiliSyncService`（IHostedService）：

```csharp
public class MeiliSyncService : IHostedService
{
    // 内存队列容量 5万，满时 DropOldest（不阻塞主路径）
    private readonly Channel<TorrentMeiliDoc> _queue =
        Channel.CreateBounded<TorrentMeiliDoc>(
            new BoundedChannelOptions(50_000)
            { FullMode = BoundedChannelFullMode.DropOldest });

    public void Enqueue(IEnumerable<Torrent> torrents)
    {
        foreach (var t in torrents)
            _queue.Writer.TryWrite(MapToDoc(t));  // 非 async，TryWrite 即返回
    }

    private async Task PushLoop(CancellationToken ct)
    {
        var batch = new List<TorrentMeiliDoc>(500);

        while (!ct.IsCancellationRequested)
        {
            batch.Clear();
            // 最多等 2 秒，或积累 500 条，取先到者
            using var cts = CancellationTokenSource.CreateLinkedTokenSource(ct);
            cts.CancelAfter(TimeSpan.FromSeconds(2));
            try
            {
                while (batch.Count < 500 &&
                       await _queue.Reader.WaitToReadAsync(cts.Token))
                    while (batch.Count < 500 && _queue.Reader.TryRead(out var doc))
                        batch.Add(doc);
            }
            catch (OperationCanceledException) { /* 2s 超时，正常 */ }

            if (batch.Count == 0) continue;

            try { await _meili.AddDocumentsAsync(batch, ct); }
            catch (Exception ex)
            {
                _logger.LogWarning(ex, "Meili sync failed {Count} docs, retrying...", batch.Count);
                foreach (var doc in batch) _queue.Writer.TryWrite(doc);  // 重新入队
                await Task.Delay(5_000, ct);
            }
        }
    }
}
```

## 第五章：统计和监控

### 5.1 三层计数器

| 层次 | 机制 | 延迟 | 精度 |
|------|------|------|------|
| 内存 | `Interlocked.Add`，每次写入时更新 | O(1) | 准确（重启归零） |
| 估算 | `SELECT reltuples FROM pg_class` | ~1ms | 误差 0.1-1% |
| 精确 | `COUNT(*)`，每小时 cron 刷新 | 数百ms~数秒 | 100% |

```csharp
public class CounterService : IHostedService
{
    private long _total, _today;
    private int  _todayDay;  // UTC DayOfYear，检测跨日

    public void Add(long n)
    {
        Interlocked.Add(ref _total, n);
        var day = DateTime.UtcNow.DayOfYear;
        if (day != _todayDay) { Interlocked.Exchange(ref _today, n); _todayDay = day; }
        else Interlocked.Add(ref _today, n);
    }

    public async Task StartAsync(CancellationToken ct)
    {
        // reltuples 快速估算总量，today 精确查（当日量小，可接受）
        var approxTotal = await repo.GetApproxTotalCountAsync(ct);
        var today       = await repo.GetTodayCountAsync(ct);
        Sync(approxTotal, today);
    }
}
```

5. Problem Solving:

**Root cause of disk IO crash identified and fixed:**
The `seenSet` type used `map[string]time.Time` with unbounded growth. Under high DHT traffic (thousands of packets/sec), with 30-minute TTL for metadata requests, the map could accumulate 1M+ entries (~400MB). Go's GC under memory pressure triggers swap activity, causing massive disk IO that could crash cloud VMs. Three instances of this problem existed: `infohashSeen`, `metadataRequestSeen`, `metadataResultSeen`. Additionally, `remoteKnown sync.Map` never cleaned up, and `wire.queue syncedMap` never deleted completed entries.

**Solution**: Replace all unbounded maps with bounded LRU caches (500k–1M capacity). Add adaptive memory monitoring that pauses metadata queuing and forces GC when heap exceeds 500MB.

**Wire.Run() efficiency fix:**
Original code had `default: continue` which silently dropped metadata requests when worker goroutines were saturated. Under high DHT traffic, this meant most discovered info_hashes never got metadata downloaded. Fixed by using blocking semaphore (`wire.workerTokens <- struct{}{}`), which provides natural backpressure from the workers channel to the requests channel.

```go
// refreshCrawlNeighborCache precomputes K-nearest compact node infos every 2s
func (dht *DHT) refreshCrawlNeighborCache() {

func handleResponseCrawl(dht *DHT, addr *net.UDPAddr, response map[string]interface{}) bool {
	if err := ParseKey(response, "r", "map"); err != nil {
		return false
	}
	r := response["r"].(map[string]interface{})
	id, ok := r["id"].(string)
	if !ok || len(id) != 20 {
		return false
	}
	// Insert compact nodes from response into routing table
	if nodesStr, ok := r["nodes"].(string); ok && len(nodesStr) > 0 && len(nodesStr)%26 == 0 {
		for i := 0; i < len(nodesStr)/26; i++ {
			no, err := newNodeFromCompactInfo(nodesStr[i*26:(i+1)*26], dht.Network)
			if err == nil && !dht.blackList.in(no.addr.IP.String(), no.addr.Port) {
				dht.routingTable.Insert(no)
			}
		}
	}
	// Process peer values via crawlTxBuf ring buffer correlation
	if dht.OnGetPeersResponse != nil {
		if values, ok := r["values"].([]interface{}); ok {
			t := response["t"].(string)
			idx := crawlTxIdx(t)
			ih := dht.crawlTxBuf[idx]
			if ih != [20]byte{} {
				infoHash := string(ih[:])
				token, _ := r["token"].(string)
				for _, v := range values {
					peer, err := newPeerFromCompactIPPortInfo(v.(string), token)
					if err == nil {
						dht.OnGetPeersResponse(infoHash, peer)
					}
				}
			}
		}
	}
	if node, err := newNode(id, addr.Network(), addr.String()); err == nil {
		dht.routingTable.Insert(node)
	}
	return true
}

	switch len(t) {
	case 0:

	case 1:
```

### 问题根因：每请求 O(n×logK) 在热路径上

**每一个 get_peers/find_node 入站请求执行路径：**

```
getPeersType (crawl mode)
  → newBitmapFromString(infoHash)        // bitmap 分配
  → GetNeighborCompactInfos(id, 8)
      → AllNodes()                        // RLock + 50k 节点遍历 + 50k 指针 slice 分配（~400KB/次）
      → getTopKNodes(50k nodes, id, 8)   // O(50k × log8) ≈ O(150k) XOR 比较
      → 收集 compactInfo 为 []string     // K 次分配
  → strings.Join()                       // 再一次分配
```

**数量化影响（按 log 估算）：**

- 日志显示 `handled=188167/30s ≈ 6.3k handled/sec`（非峰值时也有此规模）

## 流量骤降的真实原因

我之前说"可能是云服务商限速"是有根据的推测，但不够准确。结合你的具体情况，更精确的解释是：

**DHT 网络的自然平衡机制触发了**。

DHT 网络里的每个节点都维护自己的路由表（K-bucket）。当你的节点响应更快、更活跃，其他节点会把你记录进它们的路由表，并持续向你发送查询 —— 这是流量上升的原因。

问题在于，很多公网节点有简单的速率保护：**你在短时间内收到的请求超过某个阈值，对方会把你的 IP 从路由表里踢掉**（或者停止主动联系你）。这批节点在大约 50 分钟的滞后期后（DHT K-bucket 的刷新/淘汰周期通常是 15 分钟，累积效果需要几个轮次）集中体现，表现为"突然骤降"而不是渐进下降。

具体链路：

```
优化后发包率上升 (40k/s)

    → 节点的 ping 超时检测或速率限制触发
    → 这些节点不再主动发 find_node / announce_peer 给你
    → ~50分钟后（K-bucket TTL 到期 × 多轮）批量失效
```

```
crawler-3  announces(A) → Wire.Request(A) → TCP download (重复!)

shared LRU: Set(A) = true (first time)
```

### Claude

`network_mode: host` 只绕过了 Docker 的 NAT 层，但内核的 conntrack 模块对所有进出流量仍然生效。20 个 DHT 节点 × 每秒数百 UDP 包 × 30 秒 UDP 超时 = 轻易几十万条记录，默认 131072 的上限根本不够。

最彻底的解法是用 iptables `NOTRACK` 让内核直接跳过对这些端口的连接追踪：

**在宿主机上执行（立即生效）：**
```bash
# 对 DHT 端口完全跳过 conntrack（收发两方向）
iptables -t raw -A PREROUTING -p udp --dport 20003:20022 -j NOTRACK
iptables -t raw -A PREROUTING -p udp --sport 20003:20022 -j NOTRACK
iptables -t raw -A OUTPUT    -p udp --dport 20003:20022 -j NOTRACK
iptables -t raw -A OUTPUT    -p udp --sport 20003:20022 -j NOTRACK
```

**同时调大上限并缩短 UDP 超时（双保险）：**
```bash
sysctl -w net.netfilter.nf_conntrack_max=2097152
sysctl -w net.netfilter.nf_conntrack_udp_timeout=10
sysctl -w net.netfilter.nf_conntrack_udp_timeout_stream=10
```

**写入 `/etc/sysctl.d/99-dht.conf` 使重启后持久化：**
```
net.netfilter.nf_conntrack_max = 2097152
net.netfilter.nf_conntrack_udp_timeout = 10
net.netfilter.nf_conntrack_udp_timeout_stream = 10
```

**保存 iptables 规则：**
```bash
# Debian/Ubuntu
apt install iptables-persistent -y && netfilter-persistent save

# CentOS/RHEL
service iptables save
```

---

**原理说明：**

| 方案 | 效果 |
|------|------|
| `network_mode: host` | 绕过 Docker NAT，但 conntrack 仍在工作 |
| 增大 `nf_conntrack_max` | 治标，只是把上限抬高，仍然浪费内存追踪无意义的 UDP |
| 缩短 UDP timeout | 从 30s 降到 10s，条目数降低 3 倍 |
| **NOTRACK（推荐）** | 内核在 raw 表最早阶段就跳过这些包，完全零 conntrack 开销，是 DHT 这类无状态 UDP 协议的正确处理方式 |

NOTRACK 之后不仅错误消失，DHT 的包处理延迟也会略有改善，因为少了一层内核状态机的判断。
